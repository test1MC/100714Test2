Accounts Payable

A/P Recurring Invoice Maintenance

 Last Revised: 01/03/14

This function is used to set up or maintain those invoices which must be paid regularly, but for which you receive no written reminders or bills.   This eliminates the need to enter these payables repeatedly (in Invoice Data Entry) each time they are due (monthly, semi-monthly, weekly, bi-weekly).

Examples of these types of payables include rent, utilities, royalty payments, or dividends to stockholders. Once these invoices are set up, they can be placed in a batch for processing through the menu option called Recurring Invoice Generation.

You will be given the option to select from a list of all existing recurring payables, which will be displayed by vendor or to enter a new/maintain existing recurring invoices.

Note:
3 Locations will NOT appear on the A/P Invoice Data Entry Screen if the Location Segment in General Ledger Parameter Maintenance is not set to 2 or above (multiple locations).  In that case (i.e., your company operates from one, not multiple locations) only 16 menu items will appear on the A/P Invoice Data Entry Screen.
1 Vendor

Enter vendor number, alpha sort name, contact first or last name, phone number, or zip code.   Entering a single-character vendor number is not valid since this type of vendor number represents a category of individual vendors (i.e. one-time vendors).

2 Invoice #

Since an invoice number does not usually exist for recurring payables, enter a generic term or description (e.g. rent, loan).

3 Location

This entry is applicable only for those companies who have multiple locations. Location codes previously set up in G/L Location Maintenance on the GLU menu are valid entries. Default location codes for vendors are set up in VML-A/P.

When the invoice amount is distributed to the General Ledger (at the bottom of the Invoice Data Entry screen), the account number picks up the location code entered in the location field. For example, if the location code entered is 010 Akron and the invoice amount is distributed to the inventory account 4500, the G/L distribution shows 4500-010.

4 Starting Invoice Date

The starting invoice date is the date this invoice was first generated (e.g. the first date of a loan payment).

5 Terms

Terms will normally be Net upon Receipt (code 01), but any valid terms code may be used, including those that use cash discounts or multiple payments.

6 PO#

The purchase order number does not normally apply to recurring invoices but may be used if applicable.

7 Invoice Amount

This is the total amount of the periodic payment.

8 Frequency

This indicates how often you want an invoice to be generated. You will be given the option of choosing weekly, bi-weekly (every 2 weeks), semi-monthly (twice a month), monthly, quarterly, or annually.

9 Maximum Payments

This field acts as a counter to track the number of payments you wish to make. The invoice will no longer be generated after this number of payments has been reached.   (e.g. Loan amount was $100 and you wish to make 10 payments.)   If you want the invoice to continue without limit (e.g. rent) then leave this field zero.

10 Maximum Amount

This is the amount, when reached, at which invoices will no longer be generated. To avoid overpaying, the maximum amount must be a multiple of the invoice amount.   The computer will not calculate fractions of an amount.   For example, if the invoice amount is $100, the maximum amount should be a multiple of 100 such as 1,000 or 10,000.   If invoicing should continue without limit, leave the maximum amount zero.

11 Note

Any note/comment entered will print on the check stub.

12 Payment Method

Optional entries are:

  • blank = normal check

  • T = Electronic Transfer of Funds

  • W = Automated Withdrawal

  • S = Separate Check.  

For further details on these options, refer to Invoice Data Entry - Payment Method documentation.

13 1099 Form/Box

For optional entries, refer to documentation for Invoice Data Entry - 1099 Form/Box or follow the messages displayed at the bottom of the screen.

The next 3 fields are updated by the computer, but can be manually overridden if necessary.
14 Last Invoice Date

This is the invoice date on which the last invoice is generated.   It is used to determine the invoice date for the next invoice generated.   If not specified, then the invoice date for the next invoice generated will be the starting invoice date specified above.

15 Payments to Date

This displays the number of payments that have been generated. Usually, invoices are generated and paid on the same day. In this case, this reflects the number of payments that have been generated and paid.   However, if an operator generated recurring invoices and postponed actually paying the invoices, a more accurate description of this field would be number of payments that have been generated (the most recent not having yet been paid).

16 Paid to Date

Paid to Date reflects the total amount of all recurring invoices that have been generated (not necessarily paid). The same explanation applies as in Payments to Date.

These fields can be manually overridden.   This would be necessary if these recurring invoices were added to a batch which was subsequently deleted.   These fields would have to be adjusted in order to back-out the deleted recurring invoices.

17 G/L Account Description

Functions the same as in Invoice Data Entry.
