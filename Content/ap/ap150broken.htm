Accounts payable payment Processing

Manual Payment Entry

 Last Revised: 04/23/14

Manual Payment Entry is used to enter the following transactions:

  • Print one check quickly

  • Record a payment that does not require a printed check, such as a hand-written check or a payment made online or automatically deducted from your checking account

  • Reverse a previously updated payment

  • Void a check not used

  • Transfer funds from one of your bank accounts to another, either by check or online transfer

  • Import an automated payment such as one made by a payroll service using an import file

Manual Payment Entries are grouped into batches. Multiple batches may be opened at the same time if necessary. All transactions within a batch must have a payment date within the same G/L accounting period.  All combinations of entry types are permitted within the same batch.

 

 

  1. Print check
  2. Record payment
  3. Re-print lost or damaged check
  4. Reverse payment
  5. Void check not used
  6. Transfer funds
  7. Import automated payment
  8. Modify/Delete transaction in this batch
Print check

This option is used when it is necessary to print a single check quickly. You can print a check for existing invoices previously entered using Invoice Data Entry or for new invoices not yet entered.

Vendor

Select the vendor to be paid by entering the Vendor#, a few characters of their alpha sort name, a contact name, contact email address, phone#, or zip code.

Checks can also be printed for Miscellaneous vendors.  A Miscellaneous vendor is one the you do not want to set up as a permanent vendor using Vendor Maintenance.  A vendor must be previously defined in Vendor Maintenance using a single character Vendor# such as "M".

When a check is printed for a Miscellaneous Vendor, you will be prompted to enter the vendor name and address.  You can also select the name and address from prior miscellaneous invoices for this vendor.
Payment Method Remains blank for this option, indicates that a check will be printed
Check Date Just touch enter to accept today's date or enter the date to print on the check.  Note that all checks in the batch must be within the same G/L accounting period.
Bank Code Select from list of active bank codes established using Bank Code Maintenance (APU).
Check# Just touch enter to accept the next unused check number, or enter the actual check# pre-printed on the check to be used.

  1. Select existing invoice for payment: Use this option to select an existing open invoice to be paid.
  2. Create new invoice: Use this option to create a new invoice.
  3. Select existing invoice from vendor using this A/P Vendor#: Use this option when another vendor has this vendor as its A/P Vendor#.  Refer to Vendor Maintenance for additional details about the A/P Vendor# field and how it is used.

You will be prompted to select a printer.  You can place a check face up in the fold-down tray of a laser printer.  If printed to an ink-jet printer, the box borders and box shading will not be printed.  Dot matrix printers are not supported.  Contact Excellware for a quote for a fully functional, low cost, Hewlett Packard or Brother laser printer priced around $250.

Record payment

This option is used to make a payment when a printed check is not required.  This would include recording a handwritten check, recording a payment made online, or recording a payment that was automatically withdrawn from your bank account.

Vendor Select vendor as described above in 'Print check' section.
Payment Method
blank a handwritten check was used for payment
T you initiated the funds transfer online
W the vendor initiated an automated withdrawal to your bank account
Check or Payment Date Just touch enter to accept today's date, or enter the actual payment date.  Note that all payments in the batch must be within the same G/L accounting period.
Bank Code Enter Bank Code or select from list of active Bank Codes established using Bank Code Maintenance (APU).
Check# Just touch enter to accept the next unused check number, or enter the actual check# pre-printed on the check to be used.

The process to select an existing invoice, create a new invoice, or select an invoice from a vendor with an A/P Vendor# is the same as described above when printing a check.

Reverse payment

If the operator no longer wants to make a certain payment, it should be deleted before updating.

This option is used to reverse a payment previously updated using Check Update or a Manual Payment Update that has not yet cleared the bank using Bank Reconciliation.

You might need to reverse a payment if:

  • an invoice paid was for a different vendor
  • you have stopped payment on the check
  • you have decided not to pay one or more of the invoices at this time
  • an amount was entered incorrectly, etc.

Once a payment is reversed, any invoices paid will be re-opened and can be paid again later.  Use A/P Invoice Data Entry or A/P Invoice Maintenance if you need to adjust or reverse completely any invoices on the payment.

Note that Vendor Inquiry (A/P Checks option) will show the original payment as well as the reversed payment, coded with an "R" in the Pmt column for Reversal.

Why doesn't the payment I want to reverse appear in the list of reversible payments?

When the payment has already cleared the bank, it cannot be reversed in the same way as it is done before Bank Reconciliation.

To rectify this situation, the operator must complete the following in a single manual payment batch:

  1. Enter a new invoice to the incorrect vendor with a negative amount and process it as a payment (it cannot be processed as a check since checks cannot be negative).
  2. Run Manual Payment Update.
  3. Enter a new invoice to the correct vendor with a positive amount and process it as a payment.
  4. Run Manual Payment Update.
  5. In Bank Reconciliation, mark the two new payments as cleared. They will offset one another.
Void check not used

Note that this is different from a payment reversal.

This option is used whenever there is a check number that will not be used -- for example, if you print a batch of checks, and before updating discover one check is incorrect, and then delete the payment using either A/P Check Maintenance or Manual Payment Entry.  The check# originally printed should be destroyed.  In this case you should use this option to record the fact that this check# will never be used.  Voiding a check is not required, but many accountants want to account for every sequential check#.

This option could also be used if a check was damaged or otherwise not used.

This option is used to transfer funds from one of your bank accounts to another.

You have the option to print a check, or to record a funds transfer made online.

blank Use this option to print a check that you can deposit or mail to the financial institution.

W Automated Withdrawal - Use this option when you have scheduled an automated transfer that is initiated by the financial institution

Import automated payment

This option is used to record a payment where you have payment data in a file, such as what might be provided by a payroll service.  Since a payroll posting often includes many G/L account numbers and amounts, importing a payment from a file can save time and reduce data entry errors.

The import file must be a comma or tab delimited file and stored on the Dynamo server.  You can access the default folder from a Windows PC by clicking Start, Run and entering \\server\XX-shared where XX is your two letter company code.  Contact Excellware support if you cannot view this folder from Windows. 

If using Microsoft Excel, open the spreadsheet file and click File, then Save As.  Change the 'Save as type' to 'Text (Tab Delimited) (*.txt)'.  This will save the spreadsheet file on the server in a format that can be used by Dynamo.

Prior to importing a payment for a vendor, use Vendor Maintenance to specify the import column letters that will contain the G/L Account# and Amount.  You may also specify the Payment Date column, and optionally a Payment# column, useful when the file contains multiple payments, such as a payroll file that has one payment for net checks on one date, and taxes on another.  See Vendor Maintenance, Manual Payment Import view for additional details.

Import File Contents
G/L Account# The Dynamo G/L Account# with or without hyphens.
Amount Positive amount indicates debit to the general ledger, negative indicates a credit.  The sum of all the amounts for each payment need to add up to zero.
Payment# Up to 3 characters or digits.  Only required when the import file contains multiple payments.  For example, if the payroll service makes two separate withdrawals from your bank account, then each withdrawal should have a unique payment#.  These could be N for Net Pay, T for Taxes, or 1 and 2, or any other identifier to identify the payment.
Payment Date Optional.  If provided it can be in MM/DD/YY or MMDDYYYY or YYMMDD or YYYYMMDD format.  If not provided, the operator will be prompted for the payment date.

Note that Dynamo determines the bank account based on the G/L Account linked to the Bank Account in Bank Account Maintenance (APU).  The operator will be prompted should multiple active bank accounts be linked to the same G/L Account#.

You have the option to delete the imported data file after a successful import.

The following error message can occur during the import process.  Open the import file using Microsoft Excel or Open Office to correct the date in the import file if required.  Contact the creator of the file should permanent corrections be required.

Import Error Messages
Row xxx contains yyy columns but requires zzz columns Row xxx does not include sufficient number of columns to obtain the data.
Row xxx contains invalid G/L Account# of yyyy. The G/L Account number is all blank, all zeroes, contains a blank, or is longer than 9 digits, excluding any dashes.
Row xxx has an invalid payment# of yyyy. If used, the payment# must be 3 or fewer characters.
Row xxx has an invalid payment date of yyyy. If used, the payment date must be one of the formats shown above.
Row xxx has an invalid amount of xxxxx. Negative amounts must have a leading - sign.
Row xxx has a G/L Account# of yyyy that is not on G/L Accounts File Either the G/L Account# in the import file is incorrect, or needs to be added using G/L Account Maintenance (G/L).
Row xxx has a G/L Account# of yyyy that is not on G/L Locations File For companies with multiple locations, the segment of the G/L Account# that contains the Location, this error message indicates the Location is either incorrect or needs to be added using G/L Location Maintenance (GLU)
The sum of the amounts of one or more payments do not add up to zero. The sum of the amounts of each payment must add up to zero.
No G/L Account was found that is linked to an active bank account. The import process posts a payment to a bank account as determined by the G/L Account#.  Verify that the G/L Account# linked to the bank account using Bank Account Maintenance (APU) is correct and contained within import file.
Payment contains multiple G/L Accounts that are linked to a Bank Code. There can only be one G/L Account per payment that is linked to a Bank Code.

Imported payments may be viewed, modified, or deleted using the option to Modify/Delete transaction in this batch, then selecting a transaction type with 'Record' in the Trans column for the desired vendor shown.

Modify/Delete transaction in this batch

Once any of the above transactions are made, use this option to modify or delete a transaction prior to updating.

After all Manual Payment entries have been made, print the Manual Payment Register and review for accuracy.  Return to Manual Payment Entry to make any corrections required.  File the Manual Payment Register in the Dynamo Reports binder and update the transactions using Manual Payment Update.
